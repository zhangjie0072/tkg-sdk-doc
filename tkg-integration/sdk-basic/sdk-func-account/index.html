Account system

tip

Before integrating the account system, please confirm the timing of its appearance and choose the login methods you want to support. We offer guest login, email login, and third-party login methods.
Besides account login, we support account deletion (required), login type inquiry (required), logout,  account binding, and forced binding, auto login, returns the currently user login type, Return to whether the login interface needs to be displayed.
Touka SDK only provides the login interface, the login screen can be designed and created by developers according to the game's UI style.

Login methods​

  1. Other account system logins: Guest login, Email login.
  2. Third-party logins: Google login (Android only), Facebook login, Apple login (iOS only).

Account system display timing​

  1. After entering the game, the login screen is displayed directly. Players must complete the login by filling in/selecting the information to enter the game. （Only show game login screen for the first time, and after times use the selected method to log in.）

  2. Players can enter the game immediately with guest silent login and bind their accounts at a suitable game position. （non-guest account enters game, the binding button don't display.）

Explanation​

  1. Different UserIDs are returned for each login method: guest, Google, Facebook, Apple, and email.（The situation is unique when it comes to account binding.）
  2. Note that login is called through the login interface, and after successful login, the binding of a guest account to a third-party account is done through the account binding interface. Only when the account binding interface is called will an internal binding operation for the guest account and third-party account be performed in the SDK.
  3. After successful login of non-guest account types, no binding function is provided (i.e. the binding button is not displayed). After a successful guest account binding, the returned login type corresponds to the type of the third-party account that was bound, and the returned UserID is the same as the UserID returned during guest login.

Account login​

TKGSDKManager.Instance.Login(LoginType.LOGIN_BY_GOOGLE, (loginStatus,  loginType,  loginMsg,  userId,  token,  email,  nickName,  iconUrl) =>{

    Debug.Log("Login status code:"+loginStatus);
    Debug.Log("Login type:" + loginType);
    Debug.Log("Login status information:" + loginMsg);
    Debug.Log("User ID:" + userId);
    Debug.Log("token:" + token);
    Debug.Log("email:" + email);
    Debug.Log("nickName:" + nickName);
    Debug.Log("icon url:" + iconUrl);
    
});

Account delete (Required)​

To enhance transparency and user data control, Google requires games with account login to offer account deletion.

  • Games must add a deletion button, clicking it invokes the SDK's deletion interface, no further action is needed.
  • The SDK then handles deletion, provides a callback to the game, and exits immediately. Reopening the game will resemble a new user entry.
  • The deletion flow includes: long-press 'OK' for 5 seconds until progress bar fills up, triggering a second confirmation. Click 'OK' to send the game callback, closing the game; 'Cancel' exits the deletion interface.
TKGSDKManager.Instance.DeleteAccount();

Account Binding (Optional)​

Explanation
  1. The account binding function is used to bind guest accounts to third-party account services in the game.
  2. Only guest accounts can be bound to third-party accounts. Once a guest account has been bound, it cannot be bound again. If the binding interface is called again, a 'already bound account' status code will be returned.
  3. The game may hide the binding button after binding. The binding button should only be displayed when loginType in the callback is the guest login type.
  4. After a guest account is bound to a third-party account, the returned UserID is still the UserID of the guest account; it's just that the two are bound together in SDK internally.
  5. As long as the guest account hasn't been bound to a social account, it has the right to bind to a social account. If the social account has already been bound by another guest account, there is the right to force binding. This involves the forced binding interface.
  6. When users use a social account that has been previously bound for the binding operation, the game can pop up an interface listing the two accounts and corresponding game progress for players to choose which account to bind. After the player selects, calling the force binding interface can be done.
  7. After successful binding, the game needs to determine whether the UserID returned after binding is consistent with the currently logged in UserID. If it is not consistent, it needs to switch to the corresponding game progress (account switching) for the UserID returned after successful binding.

TKGSDKManager.Instance.BindAccount(LoginType.LOGIN_BY_FACEBOOK,(bindAccountStatus,loginType,loginMsg,userId,token,email,nickName,iconUrl,userIdList)=> {

Q&A​

1. Can one Google account correspond to multiple game accounts? No. One Google account can only correspond to one game account.

2. If guest login is called first, then Google login, will guest and Google accounts bind?No. This procedure will return a new UserID for a new account. Binding only occurs when the binding interface is called, returning the same UserID, specifically the guest account's UserID, using the same gaming process. However, guest UserIDs are device-dependent. If the device ID changes, the unbound guest account could be lost.

3. If I log in as a guest and return UserID1, then choose to bind the Google account after making progress, will my UserID change?
No. After binding, it will remain as UserID1. The SDK only performs an internal binding while keeping the original guest UserID.

4. If I choose Google login when entering the game, make progress, then re-enter and choose guest login, then bind the Google account, what will happen?
When binding a Google account with game progress, the SDK will return two UserIDs to the game: the current guest account UserID and the UserID associated with the Google account. The game can produce an interface for the player to choose which game progress UserID they want to bind to their Google account, which is then passed to the SDK for force binding.

5. For games that by default use guest login, if I play and bind my only Google account on device A, then play and try to bind with the same Google account on device B, what will happen?
This situation is the same as number 4, both involve binding an existing Google account.

6. Will newly generated UserIDs be duplicated?
No. Newly generated UserIDs are unique and will not duplicate.

7、Can I perform account system verification under Unity Editor?
Yes. Unity Editor can be used for debugging the account system.

8、Can I log in to a social account under Unity Editor?
No. But it can simulate login. Find ToukaGames ->Open GM Panel on the Unity Editor toolbar to open the GM interface, select the social login channel that needs to be simulated, fill in a desired UserID, save and restart the game, and then simulate the login and bind the function.

9、Can the Android package account system be used normally?
Yes. Android can use all the functions of the account system normally. If you need to simulate login and bind social accounts, you can open the login method and set the simulated userID through the debug panel.

10、How to open the Android debug panel?
Firstly, the game needs to be connected to the 'OpenPolicyPop' interface.
Step 1: Open policy popup
Step 2: Quickly click on the Title of the compliance pop-up window 5 times, and SDK will automatically close the compliance pop-up window and display a floating small ball on the game interface
Step 3: Click on the floating ball to display some configuration information for the current game
Step 4: Long press and hold the 'Close' button in the bottom right corner to display the debug panel
